## 2. 关键设计变更详解

### 2.1 数据归属：log_llm_* 不再属于本工程

**修订前（v0.x 现状）**：

```
agent-insight 工程
├── 本地 MySQL 库
│   ├── log_llm_agent_main       ← 强类型 Entity
│   ├── log_llm_task_detail      ← 强类型 Entity
│   ├── log_llm_task_step        ← 强类型 Entity
│   └── log_llm_http_request     ← 强类型 Entity
└── 本地 Mongo（insight_*）
```

**修订后（v1.0）**：

```
llm-agent 工程（🚫 本工程永远不动）
└── llm-agent Mongo 集群（可多实例）
    ├── log_llm_agent_main
    ├── log_llm_task_detail
    ├── log_llm_task_step
    └── log_llm_http_request
        ↑ 写入
        │ 只读
        ↓
agent-insight 工程
└── 本地 Mongo（insight_*）
    ├── insight_datasource   ← 包含指向 llm-agent Mongo 的 connection
    ├── insight_table_config
    ├── insight_column_config
    ├── insight_query_config
    └── insight_query_history
```

**对老 Service 的影响（渐进改造，不一次推翻）**：

| 老 Service | 老路径 | 新路径（保留老路径作为兜底） |
|------------|--------|------------------------------|
| `LogViewerServiceImpl` | `logLlmAgentMainRepository.findByRequestId(...)` | `queryEngine.query(collection="log_llm_agent_main", filter=...)` |
| `TraceAnalysisServiceImpl` | `taskDetailRepository.findByRequestIdOrderByTaskIndex(...)` | `queryEngine.query(collection="log_llm_task_detail", sort=...)` |
| `RequestSearchServiceImpl` | 3 个 Repository 组合 | `queryEngine.batchQuery(collections=[3个], filter={request_id: x})` |
| `LlmCallAnalysisServiceImpl` | `httpRequestRepository` | `queryEngine.query(collection="log_llm_http_request", aggregate=...)` |
| `SourceViewerServiceImpl` | 不变（基于文件） | 不变 |

### 2.2 多 Mongo 数据源 + 动态切分（logicalGroup + shardRole）

**`insight_datasource` schema 扩展**：

```jsonc
{
  "datasourceKey": "llm-agent-prod-shard-1",   // 物理实例唯一
  "logicalGroup": "llm-agent-prod",            // 🆕 逻辑分组：同一组多物理实例
  "shardRole": "PRIMARY",                      // 🆕 PRIMARY / REPLICA / SHARD-N
  "datasourceType": "MONGODB",
  "status": "ACTIVE",
  "connectionConfig": { /* host/port/db/credentials */ }
}
```

**MVP 阶段只实现 3 种查询模式中的 1 种**：

| 模式 | 何时启用 | MVP |
|---|---|---|
| 单实例查询（最常用） | debug、看单个环境 | ✅ 实现 |
| 聚合查询（跨实例） | 看全公司总调用量 | ⏳ 留接口 |
| 对比查询（灰度对比） | 灰度发布 | ⏳ 留接口 |

> MVP 只做单实例查询；其它两种**只预留字段 + 接口**，不在 v1.0 投入代码。

### 2.3 MySQL/PostgreSQL 执行器降级

**修订前**（HLD §3.4）：

```java
public QueryExecutor getExecutor(String datasourceKey) {
    return switch (config.getDatasourceType()) {
        case MYSQL -> mySqlExecutor;
        case POSTGRESQL -> postgresExecutor;
        case MONGODB -> mongoExecutor;
    };
}
```

**修订后**：

```java
public QueryExecutor getExecutor(String datasourceKey) {
    if (config.getDatasourceType() == DatasourceType.MONGODB) {
        return mongoExecutor;
    }
    throw new UnsupportedDatasourceException(
        config.getDatasourceType() + " 暂未支持，v1.0 仅支持 MONGODB（详见 docs/00-revision-2026-07-03.md §2.3）"
    );
}
```

**HLD 中相关章节处理**：

| 处理方式 | 适用章节 |
|---|---|
| **简化**：删多余分支 | §3.3 Factory、§7.2 多数据源配置 |
| **保留**：作为参考 | §2.2 数据源 schema、§10 Summary |
| **移走**：未来路线图 | §3.4 MySqlQueryExecutor 完整代码 → `docs/09-future-roadmap.md` |

### 2.4 模板包设计原则

**核心**：`templates/llm-agent/` 是**纯 JSON 元数据目录**，与引擎代码完全解耦。

```
templates/llm-agent/
├── README.md                       ← 用户视角：怎么把模板接入进来
├── datasource.json                 ← 1 个 datasourceKey + logicalGroup 模板
├── tables/
│   ├── log_llm_agent_main.json     ← 4 个虚拟表
│   ├── log_llm_task_detail.json
│   ├── log_llm_task_step.json
│   └── log_llm_http_request.json
├── columns/
│   └── *.json                      ← 列配置（displayName/renderType）
├── queries/
│   └── *.json                      ← 预置查询模板
└── reports/
    └── *.json                      ← 内置报表定义
```

**导入方式**：

```bash
# 一键导入模板到 Mongo
mongoimport --uri "$MONGODB_URL" \
  --collection insight_table_config \
  --file templates/llm-agent/tables/log_llm_agent_main.json
# ... 重复
```

或者 UI 端：**Templates → Import Template → 选择 templates/llm-agent/**。

**接入新模板的预期工作量**（写一份新业务监控）：

| 步骤 | 时间 | 谁来做 |
|---|---|---|
| 写 `tables/*.json` × N | 半天 | 业务方 |
| 写 `columns/*.json` × N | 半天 | 业务方 |
| 写 `queries/*.json` × N | 1 天 | 业务方 |
| 在 UI 注册数据源 + 导入 | 10 分钟 | 业务方 |

## 4. 不在 v1.0 范围（明确划线）

| # | 功能 | 原因 | 推迟到 |
|---|---|---|---|
| 1 | MySQL/PostgreSQL 执行器 | MVP 不需要 | v1.1（如果业务驱动） |
| 2 | 跨 logicalGroup 聚合查询 | 性能/一致性边界复杂 | v1.2 |

| 4 | 鉴权/多租户/RBAC | 用户未提需求 | v1.1 |
| 5 | 自然语言生成查询 | AI 模块已存在但未集成 | v1.1 |
| 6 | 根因分析报告 | 与本 BI 解耦 | 独立迭代 |
| 7 | 删老 5 个 Service | 强制稳定性优先 | v2.0（标注 @Deprecated 后 6 个月） |
