## 7. 已知问题

> T6.3 代码审查（Phase 4 第 2 人天）。审查范围：Controller → Service → Repository 各层代码。

### 7.1 N+1 查询问题（~~高优先级~~ ✅ 已修复）

**位置**：`RequestSearchServiceImpl.toRequestSummary()`（第 143–171 行）

**修复方式**：在 `searchRequests()` 中先一次性批量收集所有 requestId，再批量执行 `findByRequestId`/`countByRequestId`/`countFailedByRequestId`，用 Map 做本地聚合，消除 N+1。

返回 100 条记录时，数据库查询次数从 300+ 次降至约 4 次（1 次分页主查询 + 3 次批量统计）。

---

### 7.2 freeSql SQL 注入风险（~~高优先级~~ ✅ 已修复）

**位置**：`SqlQueryExecutor.execute()`（第 37–39 行）

**修复方式**：新增 `validateFreeSql()` 方法，对 freeSql 执行黑名单校验，拦截 `DROP/DELETE/INSERT/UPDATE/ALTER/GRANT/EXEC/` 及分号/双破折号等危险语法。检测到时抛出 `IllegalArgumentException`，返回友好错误。

### 7.3 pageSize 无上限（~~高优先级~~ ✅ 已修复）

**位置**：`SqlQueryExecutor.execute()`

**修复方式**：新增 `safePageSize` 变量，`pageSize <= 0` 时默认 20，`pageSize > 5000` 时硬截断为 5000，防止 OOM。

### 7.4 pageSize=0 导致除零异常（~~中优先级~~ ✅ 已修复）

**位置**：`SqlQueryExecutor.execute()`（第 67–68 行）

**修复方式**：`safePageSize <= 0` 时直接设为 20，消除除零可能。

---

### 7.5 discover 数据源解析不一致（低优先级）

**位置**：`SqlQueryExecutor.discoverTables()` / `discoverColumns()` / `resolveDatasource()`（第 138–143 行）

**问题**：`resolveDatasource()` 只取 cache 中第一个数据源，且 cache 是 `ConcurrentHashMap`（无序），结果不确定。

```java
private InsightDatasource resolveDatasource(DynamicDatasourceManager manager) {
    for (InsightDatasource d : manager.getDatasourceCache().values()) {
        return d;  // 返回第一个，可能不是预期的数据源
    }
    return null;
}
```

**建议**：在 `discoverTables/discoverColumns` 的请求中显式传入 `datasourceKey`，或在 `QueryRequest` 中增加 `datasourceKey` 参数。

---

### 7.6 MongoDB 外部数据源无连接池配置（中优先级）

**位置**：`DynamicDatasourceManager.createMongoTemplate()`（第 410–424 行）

**问题**：创建外部 MongoDB 数据源的 `MongoTemplate` 时，没有配置连接池参数（超时、poolSize 等），依赖 MongoDB driver 默认值。高并发场景下可能出现连接耗尽。

```java
var mongoClient = com.mongodb.client.MongoClients.create(uri.toString());
return new MongoTemplate(mongoClient, cc.getDatabase());
```

**建议**：使用 `MongoClientSettings` 配置连接池和超时：
```java
MongoClientSettings settings = MongoClientSettings.builder()
    .applyConnectionString(new ConnectionString(uri))
    .applyToConnectionPoolSettings(builder ->
        builder.maxSize(cc.getConnectionPoolSize() != null ? cc.getConnectionPoolSize() : 10)
               .maxWaitTime(Duration.ofMillis(cc.getConnectionTimeout() != null ? cc.getConnectionTimeout() : 30000)))
    .build();
var mongoClient = MongoClients.create(settings);
```

### 7.7 实体-库表 schema 漂移导致 `/api/v1/requests` 502（~~高优先级~~ ✅ 已修复 2026-07-06）

**现象**：`GET /api/v1/requests` 在生产查询列表返回 HTTP 502，前端页面空白。

**根因**：`docker/mysql/init.sql` 中三张表存在 7 个多余字段（`agent_end_time` / `agent_id` / `agent_result` / `agent_status` / `agent_try_count` / `biz_id` / `chat_message_id`），但对应 JPA 实体 `LogLlmAgentMain` / `LogLlmTaskDetail` / `LogLlmTaskStep` 未声明。Hibernate 在生成动态 update 语句时把这些字段当成"待持久化属性"，触发 `PropertyReferenceException: Not-allowed property`，最终由 GlobalExceptionHandler 转成 502。

**修复方式**：
1. 删除 `init.sql` 中多余的 7 列（保留运维自管的真正业务字段）
2. 实体文件移除 `// 暂时没有 agentEndTime 字段` 类注释干扰
3. 新增 `EntitySchemaContractTest`（位于 `src/test/java/com/llm/insight/repository/entity/`），校验"数据库字段 ⊆ 实体属性"，作为防回归守门测试

**新增测试**：
- `EntitySchemaContractTest.verifyDatabaseColumnsMatchEntityFields`：用 H2/直连 MySQL 拉 `INFORMATION_SCHEMA.COLUMNS`，对每张 `log_llm_*` 表断言"实体声明的字段集 ⊇ 真实表字段集"
- 失败时报告具体哪个字段缺失/多余，便于快速定位

**回归触发条件**：未来谁要往 `log_llm_*` 表新增字段，**必须同步在实体类声明**，否则这个测试会立即红。
